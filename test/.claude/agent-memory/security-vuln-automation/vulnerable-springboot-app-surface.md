The OWASP lab Spring Boot app has a few non-obvious surface details that
diverged from the security-vuln-testcase-generator's expected results and
that future SEC-00X test cases need to account for:

1. **There is no Thymeleaf /register page.** Registration is API-only
   (`POST /api/register`). The only Thymeleaf form is /login. The
   `src/main/resources/templates/` directory has `login.html`,
   `dashboard.html`, `users.html`, `products.html`, `comments.html`,
   `transfer.html`, `vulnerabilities.html` — no `register.html`.
   Implication: SEC-*UI cases that target a "registration form" must
   either drive the API through the browser's request context or
   document the surface mismatch.

2. **The /login Thymeleaf page renders the seeded plaintext credentials
   (`alice123`/`bob123`/`admin123`) in a `<details>` demo-credentials
   block.** This is intentional in the lab. A `document.body.innerText`
   check for "no plaintext leak" must therefore be made on a post-login
   page (`/dashboard`, `/users`, etc.), not on `/login` itself.

3. **`/api/register` and `/api/profile/{id}` responses still serialise
   the full `User` entity via Jackson, including the BCrypt-hashed
   `password` field.** The JSON test-case file's
   `evidence.register_body_has_bcrypt_hash: false` is therefore wrong
   for this build — the hash IS in the body. The security property is
   "plaintext must not appear in the response body" (which the test
   correctly asserts). The test should not assert "no hash present" on
   register/profile/users endpoints.

4. **The 401 from `POST /api/login` with bad credentials does NOT carry
   `WWW-Authenticate: Basic`.** The AuthController issues its own
   `ResponseEntity.status(401).body(...)` — the request is past the
   auth filter at that point, so the entry-point's Basic challenge
   does not fire. The 401s that DO carry `WWW-Authenticate: Basic`
   are the ones from the content-type-aware entry point (i.e. callers
   blocked at the auth filter, not ones that reach the controller).
   Implication: only assert the Basic challenge on 401s from anonymous
   calls to PROTECTED endpoints (`/api/users`, `/api/profile/{id}`),
   not on credential-rejected 401s from `/api/login` itself.

5. **`/api/users` is admin-only — non-admin (USER) callers get 403.**
   The remediated `UserController.listUsers` requires `ROLE_ADMIN` and
   throws `AccessDeniedException` for USER-role callers. An
   `I am authenticated as "alice:alice123" / I GET /api/users` step
   therefore returns 403, not 200. Use `I am authenticated as
   "admin:admin123"` for the positive-control 200, and `one of 200,
   403` if the test wants to assert "not 401".

6. **Delegating PasswordEncoder produces `{bcrypt}$2a$10$...` prefixed
   hashes.** `PasswordConfig` uses
   `PasswordEncoderFactories.createDelegatingPasswordEncoder()`. Any
   regex the test uses for "is this a BCrypt hash?" must accept the
   optional `{bcrypt}` prefix.

**Why:** The test-case JSON files in `test/testcaseInJson/` are
generated by another agent that doesn't read source code; it works
from the assessment report. Its expected_result/evidence fields
occasionally disagree with the actual remediated app behaviour, and
this is the most common shape of that disagreement.

**How to apply:** When extending the test suite to new SEC-* findings,
before writing steps, read the matching controller, security config,
and template to confirm the surface. If the test-case JSON asserts
something the live app doesn't do, either (a) loosen the assertion
to the security property that IS preserved and add a code comment
documenting the deviation, or (b) flag the deviation in the run
report. Do NOT mark the step `pending` — that hides the gap.
